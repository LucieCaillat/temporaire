Ressource : Interface administrateur

1. Introduction

Comment mettre en place un système d'interface admin avec Rails ? Nous allons voir cela dans cette ressource.

2. Historique et contexte

Gérer la base de données avec la console Heroku est bien au début. Mais rapidement en ont marre de faire des commandes du genre project.update(validated: true). Afin d'éviter cela, des interfaces admin sont généralment ajoutées aux sites. Elle permettent de modifier ce que tu veux dans la base de données.

3. La ressource

Pour cette ressource, nous allons voir trois façons de faire une interface admin :

  • Brancher sa base de données à une SaaS qui fait tout le travail pour toi
  • Installer une gem qui va générer des scaffold administrateurs
  • Coder à la main son interface administrateur

3.1. Une SaaS

Il existe des services où tu as juste à brancher des clé d'APIs, qui auront un accès à ta BDD, et qui te vendent une interface toute faite customisable à sa guise. ForestAdmin fait exactement cela. ForestAdmin est une entreprise française originaire de l'un des startup studios les plus célèbre au monde : eFounders.

Tu as juste à installer la gem, mettre les clés d'APIs, pousser en prod, créer ton compte, puis à toi la gloire.

3.2. Une gem

Si jamais tu trouves ForestAdmin trop cher / pas assez permissif, voici une sélection de gems qui te permettent d'avoir une interface admin :

  • administrate de nos amis Thoughbot (que l'on recommande)
  • rails_admin de sferik (le créateur de la gem Twitter)

3.3. Du code

La technique que l'on utilise personnellement : coder son dashboard Admin à la main. Cela a plusieurs avantages : déjà, cela permet une maîtrise totale de ce que tu veux pouvoir y modifier. Ensuite, comme les views sont souvent des scaffolds purs et durs, et bien cela va assez vite au final. Voici quelques conseils pour faire son dashboard admin à la main :

3.3.1. /admin

Comment ranger proprement son code ? Avec la notion de namespace, tu peux facilement mettre tout ce qui concerne le dashboard admin dans un dossier bien rangé. Puis comme cela tu peux avoir deux controllers pour les users : un pour l'interface normale, puis un autre dans le namespace admin. Le premier va juste faire des opérations limitées, puis l'autre c'est open bar pour faire ce que tu veux.

3.3.2. Accès aux admins ?

C'est simple : tu fais une méthode check_if_admin en before_action dans tous tes controllers Admin. Puis tu ajoutes un attribut is_admin booléen à tes utilisateurs. Si le current_user n'est pas admin, tu l'envoies chier quand il arrive sur n'importe quelle action de ton dashboard admin.

🤓 QUESTION RÉCURRENTE

Mais dis-donc Jamy, vaut-il mieux faire un attribut is_admin aux utilisateurs, ou bien ajouter un model Admin indépendant du model User ?
Très bonne question, et il y a deux écoles qui se valent. Voici ce qui peut faire pencher la balance vers un côté ou l'autre :

  • L'attribut is_admin est bien plus rapide à coder que la génération d'un model via Devise pour les admins
  • Si tes administrateurs ne peuvent pas être utilisateurs de ton site (trombinoscope des élèves d'une école par exemple), c'est mieux d'avoir un model Admin
  • C'est légèrement plus sécurisé d'avoir le model Admin, car tu auras moins de chance de créer un model que de faire user.update(is_admin: true)

3.3.3. Le CSS ?

Pour le CSS, restons simples : tu peux utiliser les exemples de Bootstrap. Sache que l'Asset Pipeline gère les CSS spécifiques aux controllers, même si tu peux faire des miracles avec Bootstrap vanilla 🚀

3.4. Quelle solution choisir ?

Personnellement nous utilisons un dashboard admin custom. C'est extrêmement simple à mettre en place avec l'attribut admin du model User, puis il suffit juste de créer les views (ça va assez vite). Nous avons fait ce choix car :

  • ForestAdmin est simple à mettre en place, mais rapidement limité dans sa version gratuite
  • On nous a dit le plus grand mal de ActiveAdmin, qui est la gem la plus célèbre pour gérer une interface admin

4. Points importants à retenir

Tu as trois façons pour gérer une interface admin dans une app Rails :

  • ForestAdmin
  • Une gem (administrate)
  • Un dashboard admin codé à la main

1. Introduction

Dans cette courte ressource, nous allons voir comment faire des jolis fichiers READMEs pour donner envie à ses lecteurs.

2. Contexte et historique

Le .md correspond à l'extension d'un fichier markdown, qui permet de faire du texte riche (texte avec du formatage). L'avantage du markdown est que c'est un format très simple à écrire et il contient tout ce dont nous avons besoin pour écrire du texte avec du formatage basique. Ce format est tellement simple, qu'il a été adopté, en même pas 15 ans d'existence, par GitHub, Reddit, Wordpress, Trello et bien d'autres. Maîtriser markdown permet d'être facilement compris par tout futur collègue lors de ta vie professionnelle

En gros, pour écrire quelque chose comme un cours, un fichier markdown est le format idéal. Tu peux trouver par exemple des exemples de markdown stylays ici ou là

Projets : Création d'un interface administrateur à Evenbrite

1. Introduction

Dans cet exercice final du Eventbrite, tu vas implémenter la possibilité de gérer les utilisateurs et les événements à partir d'un dashboard admin. Avec ceci, ton application sera prête à être vendue aux plus offrants 🤑

Voici donc ce que tu vas faire :

  • Coder à la main un dashboard admin
  • Insérer la possibilité de gérer les utilisateurs à partir de ton dashboard admin
  • Coder une fonctionnalité où l'administrateur du site pourra valider les événements avant de les voir sur le site
  • Insérer la possibilité de gérer les événements à partir de ton dashboard admin

2. Le projet

Pour cet exercice, nous allons te demander de coder à la main le dashboard administrateur. Tu vas voir c'est simple 😉

2.1. Un dashboard admin

Ajoute un namespace admin, puis fais à l'intérieur une root qui va à admin#index. Ce sera la page d'accueil de ton dash admin, et elle souhaitera la bienvenue à ton administrateur. Les controllers administrateurs ne devront être accessibles qu'aux administrateurs de ton application.

2.2. Gestion des utilisateurs

Tu peux faire un CRUD complet des utilisateurs à partir du dahsboard admin : les afficher, les modifier, les supprimer, etc.

2.3. Validation d'événements

La partie un peu plus intéressante. Voici la fonctionnalité que tu aimerais bien implémenter :

  • À partir de maintenant, quand un utilisateur créé un événement, ce dernier sera "en attente de validation". Tant que l'événement n'est pas validé, il ne sera pas en ligne sur le site
  • À partir du dashboard, il est possible de voir tous les événements qu'il faut valider, puis de les valider un par un (ou pas)
  • Si un événement est validé (ou pas), l'utilisateur reçoit un email pour l'avertir de ceci. Puis il sera possible de voir l'événement sur la partie normale du site

Comme c'est la première fois que tu fais ceci, l'exercice sera un peu plus pas à pas. Nous allons te montrer comment faire des models "en attente de validation", ce qui est une fonctionnalité assez indispensable !

2.3.1. Backend des validations

Il y a plusieurs façons astucieuses pour ceci :

  • Ajouter une colonne dans la table events :
    • Une colonne validated, qui est un booléen qui dit si l'événement est validé ou pas. Le booléen est nil par défaut. Il passera à false si l'événement est refusé, et à true s'il est accepté. Dans les views utilisateurs par exemple, @events ira prendre que les événements qui sont validés
  • Ajouter deux colonnes dans la table events :
    • Une colonne validated, qui est un booléen qui dit si l'événement est validé ou pas. Le booléen est false par défaut. Il passera à true si l'événement est validé
    • Une colonne reviewed, qui est un booléen qui dit si l'événement doit être relu ou pas par l'administrateur. Il est false à la création (il doit être revu), et passera à true une fois revu
  • Créer une table event_submissions qui est une proposition d'événement et qui s'occupera de table intermédiaire

Je te laisse prendre la solution qui te va bien puis de changer les models ♥

2.3.2. Changement dans l'interface normale

Dans l'interface normale, il y a quelques détails à changer :

  • La création d'un événement va changer
  • L'index des événements va changer
  • Impossible de faire #show sur un événement en cours de validation

2.3.3. Interface admin

Puis dans l'interface admin, il te faudra faire une view avec l'index des événements à valider. Tu peux le faire avec une resource fictive, telle que event_submissions.

🤓 QUESTION RÉCURRENTE

Mais dis-donc Jamy, peut-on prendre la resource events ?
Pas vraiment, car quand tu vas gérer les événements à partir de l'interface administrateur, tu utiliseras le controller events. C'est mieux donc d'utiliser un controller adapté à ceci : un controller pour les event_submissions (même si tu n'as pas créé de model EventSubmission, c'est juste que l'on les appelle comme ça)

Tu aurais donc :

  • La méthode index, qui recense tous les événements à valider
  • La méthode show, qui affiche un événement à valider précis
  • Les méthodes edit/update qui permettent de valider (ou pas l'événement)

2.4. Gestion des événements

Enfin, pour ne pas perdre la main, fais pour les événements ce que tu as fait pour les utilisateurs.
